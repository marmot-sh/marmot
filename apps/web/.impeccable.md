# Impeccable — Design Context for `@marmot-sh/web`

This file is read by the impeccable skill (and related design skills) before any
design work. It captures intent that cannot be inferred from the codebase.

## Design Context

### Users

Two overlapping groups, both fluent in the shell:

1. **Seasoned terminal users** — engineers, ops/SRE, infra folks who live in
   their shell, pipe everything, and have strong opinions about CLI ergonomics.
   They are skeptical of marketing fluff and reward precision.
2. **AI-agent operators** — people running Claude Code, Codex, and similar
   agents who are stitching AI into pipelines and scripts. They want predictable
   JSON, streaming, and tools that compose with `|`.

Both audiences read landing pages with high suspicion. The page must earn trust
in the first viewport with substance, not stock phrases.

### Brand Personality

Three words: **fun, cool, edgy.**

This is not a sober enterprise dev-tool. It is opinionated, has a voice, and is
willing to break grid for a moment of personality. But it stays *precise* —
edgy means confident type and unexpected composition, not cluttered or chaotic.

Tone of voice should feel like a senior engineer who is good at their craft and
enjoys it — terse, occasionally dry, never corporate.

### Aesthetic Direction

**Reference points** (the things that capture the right feel):

- **Drizzle** — bold typographic identity, willing to use big color, has a
  visual point of view, treats type as a graphic element rather than just text.
- **PlanetScale** — editorial discipline, careful typography, monospace
  eyebrows, structured hierarchy, restraint between the loud moments.
- **PostHog** — irreverent, blocky, illustrative warmth, doesn't take itself
  too seriously. Comfortable being a *brand* and not a template.

**The blend**: PlanetScale's editorial structural rigor as the *default*, with
Drizzle-grade typographic confidence in hero/section moments, and PostHog's
willingness to have personality in the small things (footers, error states,
empty states, 404s, hover affordances).

**Anti-references** (explicit DO NOT):

- Generic SaaS landing template (icon-with-rounded-corner + heading + 3-col
  feature grid).
- Cyan-on-dark cyberpunk dev-tool aesthetic with neon glow accents.
- Glassmorphism and frosted cards used decoratively.
- Notion-clean / Linear-clean aesthetic that has no point of view of its own.

**Boldness dial: pushed up.** The current landing page is restrained editorial
dev-tool. Future work should push further — bigger typographic moves,
unexpected layout breaks, more personality in copy and detail. Not maximalism;
*confident* editorial.

### Theme

**Light is canonical.** The marketing screenshot, the social card, the hero
the user remembers — all light. Dark mode is fully supported (toggle exists,
tokens are defined) and must remain first-class for users who prefer it, but
design decisions should be made in light first, then verified in dark.

### Mascot

There is an **illustrated marmot mascot** (see `~/Downloads/marmot.webp` and
wherever it lives in `public/`). Hand-drawn texture, directional fur strokes,
cream belly, deep-orange fur, paws held together, friendly-but-not-saccharine
expression. Cast shadow grounding it.

The mascot is **the PostHog-style personality move** for this brand. Use it
deliberately:

- **Sparingly in chrome** — never in the global header. Header uses the flat
  `MarmotMark` silhouette only.
- **Strongly at narrative moments** — hero illustration, 404 page, empty
  states, docs landing, "thanks for installing" screens, section breaks where
  warmth is appropriate.
- **At a real size** — when the mascot appears, give it room. A 24px mascot is
  a wasted asset; this is a 200–400px+ illustration moment.
- **No effects on it** — no drop shadows added in CSS, no colorize, no hover
  bounce. It's a finished illustration; respect it.

The flat `MarmotMark` SVG (in `src/components/logo.tsx`) and the illustrated
mascot are **two different assets with two different jobs**, and they
coexist. Flat mark = chrome and favicons. Illustrated mascot = personality.

### Color

**Accent: deep orange** (the marmot's fur). Replaces the current emerald.

Two-color brand palette derived from the mascot:

```css
:root {
  /* deep orange — the marmot's fur, primary brand mark */
  --accent:      oklch(64% 0.18 42);
  /* warm cream — the marmot's belly, soft fills and tinted backgrounds */
  --accent-soft: oklch(94% 0.035 75);
  /* charcoal — the linework, strong type accents on light bg */
  --ink:         oklch(22% 0.01 50);
}

.dark {
  --accent:      oklch(76% 0.16 50);   /* lifted for dark surfaces */
  --accent-soft: oklch(30% 0.07 45);
  --ink:         oklch(96% 0.005 75);
}
```

**Neutrals stay cool (hue 250)** — the cool-vs-warm contrast between cool
gray surfaces and warm orange/cream brand color is core to the identity. Do
not warm the neutrals to "match" the mascot. The dissonance is intentional;
it's what keeps the page feeling editorial and not childlike.

The accent is **rare by design**. Use it for: brand mark moments, the prompt
glyph (`$`/`●`), one CTA per section, link hover states, selection,
mascot-adjacent moments. Body copy stays in `--fg`/`--muted`.

The cream `--accent-soft` is the right backdrop *behind* the mascot when it
appears in a hero — it echoes the belly without competing.

### Typography

Currently shipped: **Archivo Variable** (sans) + **Geist Mono Variable** (mono).
Both are good, neither is on the reflex-reject list, both stay.

Rules for this project specifically:

- Mono small-caps eyebrows with the `●` glyph are part of the brand. Keep.
- Hero headlines use fluid `clamp()` with strong negative tracking and tight
  leading (`leading-[0.96]`). Marketing pages only — app/docs UI stays on
  fixed `rem` scales.
- Mono is not a "developer vibes" prop here — it's a *functional* element for
  shell code, file paths, package names, and command output. Don't sprinkle
  mono on labels that aren't shell-adjacent.

### Spacing & Layout

- Containers use the `container-app` utility (max 80rem, 1.5rem inline
  padding). Honor it.
- Sections separated by hairline `border-[color:var(--border)]`, not cards.
  No card-in-card.
- Asymmetric two-column hero is part of the brand. Editorial, not centered.
- `border-left:` greater than 1px is permanently banned (impeccable rule, but
  doubly emphasized here — it does not fit the aesthetic).

### Motion

Subtle by default. Save motion for high-impact moments (hero load, page
transitions). No bounce easing. `prefers-reduced-motion` is honored.

Edgy ≠ animated everywhere. A still page with confident typography reads as
edgier than a page with hover wiggles on every card.

### Accessibility

- **Target: WCAG AA** for both themes. All accent usage on text must clear
  4.5:1 against its background; the orange `--accent` is borderline for body
  text on white — use it for large type, glyphs, and non-text indicators, and
  use `--fg` for body copy.
- `prefers-reduced-motion: reduce` must be respected wherever motion is added.
- Keyboard focus must be visible on all interactive elements (currently
  inherited from defaults — verify when components grow).

### Design Principles (decision filter)

Five principles future design work should pass through:

1. **Earn the first viewport.** Substance over stock. A real terminal block,
   real copy, real specifics — not vague benefit claims.
2. **Editorial first, app second.** Marketing pages compose like a magazine
   spread; app/docs UI composes like a reference manual. Don't blend the
   two — let each be itself.
3. **Type does the heavy lifting.** Hierarchy, personality, and rhythm all
   come from typography before they come from color or imagery.
4. **The accent is rare.** Deep orange is a punctuation mark, not a paint
   bucket. If everything is highlighted, nothing is.
5. **Speak like a senior engineer.** Terse, specific, occasionally dry.
   Never "elevate your workflow." Never "unleash the power of."
